## Latency layer

**Symptom**: 3.5s for `/v1/health/env-invariants`, 2s for
`/v1/integrations`, 1-3s for `/v1/users/me`. Operator-perceived: "the
app feels slow."

**Root cause** (measured via `curl -w "%{time_total}"` from inside the
backend container): Cairo→Supabase eu-west-1 RTT dominates query time.
Each round-trip is ~200ms warm, ~600ms cold. /me runs 5 sequential
queries → 1-3s. /tasks/query was running count() + all() = 2 round
trips per call.

**Fixes shipped (`d7993d0`)**:
- **`me_cache.py`** — Redis cache for `/v1/users/me` with 30s TTL.
  Cache HIT path is one Redis GET (~5ms). Bust on user-row mutations
  (~10 endpoints) + first `TaskManager.create_task` per user (only task
  mutation that flips `has_active_task_history`). Accept 30s stale on
  task counts.
- **`/tasks/query` window function collapse** — replaced
  `query.count() + query.all()` with `add_columns(func.count().over())`
  in a single SELECT. Saves one round-trip per call.

**Verified**: SSR latency for `/today` dropped 480ms → 360ms; cache hit
sub-second.

**v2 regression** (in flight): /pulse fires 5 parallel queries, the
14-day `/tasks/query` range for charts is the heaviest. Deferring the
chart queries + adding server-side cache for range queries is the next
fix.

## Bugs found + fixed

### `cd7de33` — `can't subtract offset-naive and offset-aware datetimes`

**Stack**:
```
File 'task_manager.py', line 411, in create_task
    int((created_at_ts - nudge_viewed_at).total_seconds())
TypeError: can't subtract offset-naive and offset-aware datetimes
```

**Family**: same as `LYR-019` and the H0 hotfix. Per
`docs/root_cause_analysis_2026_04_29.md` "tz drift family" — project
convention is naive UTC for stored datetimes; Pydantic deserializes
ISO strings with offsets ("...Z") into tz-aware datetimes. Any
arithmetic against `now_utc()` raises this exact TypeError.

**Pre-existing latent bug**, not introduced by /pulse. Surfaced
because the new BrainDumpQuickModal makes Capture-with-nudge trivially
repeatable.

**Fix**: targeted `strip_tz()` at point-of-use in
`TaskManager.create_task` on `nudge_viewed_at`. The boundary-
normalization sweep across all ~50-100 sites is still parked (per the
Apr 29 root-cause analysis "deferred for review").
