## Catalogue, Not Backlog

**A computation's presence in this document records a question that may become
answerable. It does not create implementation priority, product scope, model
authority, or permission to expose the result.**

## Hard Stop

This catalogue does not authorize:

- a new metric in runtime;
- a plan score or universal ranking;
- a plan-version schema;
- a decision tree, survival model, clustering model, or AI model;
- a new Pressure Map, Insights, prediction, or recovery behavior;
- automatic plan mutation;
- online learning or policy optimization;
- causal, psychological, identity, clinical, or productivity claims;
- transfer from founder history to cohort defaults.

## 1. Method-Selection Principle

Every question receives the least powerful method that can answer it honestly:

```text
canonical state and exact arithmetic
-> deterministic rules and interval operations
-> descriptive distributions and stratification
-> process alignment and survival analysis
-> shallow interpretable statistical learning
-> hierarchical or multilevel models
-> simulation and falsification
-> experimental causal methods
-> bounded reasoning-runtime support for residual semantic ambiguity
```

Later methods do not replace earlier authority. They operate on admitted,
versioned features and remain subordinate to deterministic state, exposure,
privacy, and mutation boundaries.

## 2. Three Independent Canonical Feature Families

### 2.1 Work and plan outcome family

Candidate dimensions:

- deadline displacement;
- completion and scope relation;
- active-time calibration;
- initiation displacement;
- temporal displacement and plan revision;
- interruption and open-thread footprint;
- downstream displacement;
- explicit recovery latency.

### 2.2 Observation-quality family

Candidate dimensions:

- plan acceptance/version known;
- execution provenance;
- raw versus corrected projection;
- recovery or auto-close status;
- linkage quality;
- timestamp coherence;
- censoring and missingness;
- exposure completeness;
- clean-profile eligibility;
- privacy and retention permission.

### 2.3 System burden and participation family

Candidate dimensions:

- decision opportunities;
- delivered candidates;
- authenticated browser renders;
- dismissals, acknowledgements, actions, expiries, and losses;
- interruptiveness class;
- accepted system estimates;
- accepted system plan transformations;
- resulting canonical mutations;
- exposure carryover and competing surfaces.

Observation quality and system burden must not be optimized as if they were
intrinsic user/work outcomes.

These families are not one lowest-common-denominator payload. Named consumer
projections select compatible dimensions, horizons, and aggregations for each
decision contract while preserving canonical atomic definitions.

## 3. Current Deterministic Computation Inventory

- planned task load;
- remaining demand;

Interpretation ceiling:

- approximate workload orientation;
- not exact capacity;
- not exact collision without availability;
- not user quality or stress;

### 3.2 Capture and estimate priors

Current behavior includes:

- explicit duration extraction;
- natural-language time-range extraction;
- category priors;
- fallback duration;
- duration clamps;
- deterministic category inference;
- deadline candidate scoring and confidence tiers;
- explicit user confirmation;
- idempotent batch commit.

Interpretation ceiling:

- candidate structure and planning estimate;
- not scope truth, dependency truth, or execution truth.

### 3.3 Personal calibration

Current behavior includes:

- ratio of summed executed to summed planned duration;
- mean per-session execution multiplier;
- category, time-of-day, and duration-bucket cells;
- fallback ordering across cells and research priors;
- provisional personal-history blend;
- occupancy adjustment from clean pause overhead;
- creation-nudge thresholds;
- accepted-versus-dismissed descriptive nudge outcomes.

Interpretation ceiling:

- descriptive estimate calibration under admitted history;
- not Bayesian posterior unless implemented and validated as such;
- not causal effect of the nudge;
- not stable user identity.

### 3.4 Cortex task evidence

Current canonical measurements include:

```text
P = planned active minutes
E = observed executed active minutes
W = wall-clock elapsed minutes

execution_multiplier     = E / P
log_execution_multiplier = log(E / P)
active_delta_minutes     = E - P
wall_delta_minutes       = W - P
pause_overhead_minutes   = max(0, W - E)
```

Current topology hypotheses include bounded, expanding, fragmented,
biological, and unknown classes. These remain structural hypotheses, not
psychological causes.

### 3.5 Reflection and interruption computations

Current behavior includes:

- absolute readiness/reflection difference;
- signed readiness/reflection shift;
- active versus wall occupancy;
- execution-to-wall ratio;
- median pause duration;
- micro-mirror gates for long pauses, delayed initiation, duration mismatch,
  repeated pauses, and large execution multipliers.

Interpretation ceiling:

- descriptive relation among self-report and observed timing;
- not objective focus, cognitive load, or recovery success.

## 4. Computations Possible Without A New Schema

These are analysis candidates only. They require a frozen analysis profile and
must be labeled `latest_plan_compatibility`.

| Candidate | Definition sketch | Required caveat |
| --- | --- | --- |
| Signed active error | `E-P` | Current/latest mutable plan only |
| Relative active error | `log(E/P)` | Requires positive `P` and clean `E` |
| Start lag | actual start minus current planned start | Current plan may have been edited |
| Wall overhead | `W-E` | Switch topology can double-charge occupancy |
| Deadline displacement | executed end minus due time | Completion source and due revision matter |
| Completion shortfall | `100-completion_percent` | Self-reported and optional |
| Scope relation | bounded/expanded/reduced/replaced/unknown | Depends on explicit scope evidence |
| Reschedule count | current counter | Magnitude and prior versions unavailable |
| Explicit re-entry latency | interruption/pause to recorded resolution | Censor unresolved episodes |
| Planning participation | accepted/known planned episodes over eligible work | Acceptance is incompletely recorded |
| Unplanned execution ratio | unplanned or retroactive episodes over admitted work | Source classification may be incomplete |
| Estimate-source disagreement | user, prior, semantic, provider, personal range comparison | Never stack dependent priors as independent evidence |
| Pressure-to-action funnel | compute, render, preview, confirm, mutation, later outcome | Operator diagnostic only until exposure-complete |

### 4.1 Required denominator waterfall

Every analysis should report how many rows remain after:

```text
all candidate tasks
-> known user and non-test scope
-> nonvoid and nonanchor
-> coherent plan fields
-> explicit or defensible plan status
-> observed, closed execution
-> no recovery or retroactive synthesis
-> no user correction for clean calibration
-> usable outcome window
-> exposure state known
-> analysis-specific eligibility
```

The exclusions are part of the result.

### 4.2 Falsification before prediction

Before fitting any model:

- compare ordinary plans with synthetic one-hour immediate-start plans;
- compare ordinary plans with retroactive `planned = executed` rows;
- find exact-adherence/poor-outcome examples;
- find large-deviation/good-outcome examples;
- compare summed task occupancy against interval-unioned session time;
- recompute using raw, corrected, and clean projections;
- sample plausible scalar weights and measure rank reversal;
- stratify by exposure and estimate source;
- report user and active-day support.

If artifact paths look better than clean ordinary plans, the proposed target is
invalid.

## 5. Computations Enabled By Accepted Plan Versions

The following become well-defined only after an approved accepted-plan-version
contract exists.

### 5.1 Plan revision and survival

- time from acceptance to supersession;
- number and type of revisions;
- magnitude of schedule movement;
- duration-range widening or narrowing;
- scope additions, removals, and substitutions;
- deadline rebinding;
- plan survival curve;
- censoring at unresolved or deleted plans.

Plan survival is not plan quality. Revision can be beneficial adaptation.

### 5.2 Slack and buffer dynamics

- accepted slack by obligation and horizon;
- slack consumption over execution;
- buffer exhaustion time;
- proportion of protected buffer consumed;
- pressure contribution under low/high duration scenarios;
- sensitivity to plausible delay or scope expansion.

### 5.3 Plan fragility

A plan is descriptively fragile when small plausible disturbances cause large
changes in feasibility or downstream displacement.

Candidate sensitivity analysis:

```text
for each accepted plan:
  perturb duration within accepted uncertainty
  perturb start within observed initiation distribution
  insert observed-class interruption scenarios
  preserve all hard deadlines and constraints
  recompute feasible region and displaced obligations
```

This is scenario sensitivity, not a forecast and not a psychological claim.

### 5.4 Recoverability

Candidate questions:

- what is the smallest confirmed transformation that restores feasibility?
- how much scope, timing, or ordering changes?
- which obligations remain protected?
- how many accepted commitments are displaced?
- how long until an explicit stable resolution?
- does the plan remain unresolved at the observation-window end?

Recoverability is a property of the represented plan and available actions,
not a trait of the user.

### 5.5 Cascade and downstream displacement

Candidate descriptive objects:

- obligation-to-obligation temporal displacement;
- accepted sequence changes;
- unresolved open-thread propagation;
- deadline-cluster effects;
- later skips or reschedules after a mismatch;
- exposure-conditioned versus known-unexposed strata.

Association is not proof that the earlier event caused the later event.

### 5.6 Plan transformation comparison

Candidate transformations:

- split;
- shrink;
- defer;
- reorder;
- widen start window;
- add buffer;
- remove optional work;
- protect one deadline at the cost of another;
- ask for missing scope information.

The comparison output should be a set of trade-offs and uncertainty, not an
automatic mutation.

## 6. Interpretable Model Families

### 6.1 Rule sets

Best when:

- domain invariants dominate;
- sample size is low;
- thresholds are policy choices;
- exact auditability is required.

Risks:

- hand-tuned threshold proliferation;
- hidden interactions;
- stale rules under distribution shift;
- duplicated logic across surfaces.

### 6.2 ID3-style classification tree

Useful intuition:

- derive compact conditional branches from examples;
- reveal interactions that many independent rules miss;
- produce a path that a human can inspect.

Poor fit as the default LyraOS estimator because:

- classic ID3 targets categorical outcomes;
- information gain favors variables with many possible splits;
- continuous values need discretization;
- missingness and censoring are not first-class;
- one label cannot represent the outcome vector;
- small founder datasets will produce unstable branches;
- a leaf can look like an identity label if copy is careless.

### 6.3 Shallow regression tree

Candidate use:

- one continuous outcome dimension, such as log execution multiplier or start
  lag;
- bounded depth;
- minimum user, session, and active-day support;
- chronological validation;
- leaf distributions rather than point declarations.

### 6.4 Conditional inference tree

Candidate advantage:

- separates variable-selection testing from split selection;
- reduces preference for features with many cut points;
- permits explicit statistical stopping.

It still requires independent validation, missingness handling, exposure
strata, and interpretation limits.

### 6.5 Survival tree or survival model

Candidate use:

- time to plan supersession;
- time to explicit recovery;
- time to initiation;
- time to deadline resolution.

Why survival framing matters:

- unresolved episodes are censored, not zero;
- observation windows differ;
- disappearance from the dataset is not resolution.

### 6.6 Multi-output trees

Candidate use:

- discover plan regimes associated with several outcome dimensions.

Primary concern:

- one split criterion silently chooses trade-offs among unlike outcomes.

Preferred first approach:

- separate shallow trees per registered outcome dimension;
- compare whether partitions recur;
- preserve Pareto trade-offs;
- do not collapse them until preference ownership exists.

### 6.7 Reviewed rule distillation

A model may be used offline to propose a compact rule candidate:

```text
model discovers branch
-> analyst checks stability and artifacts
-> branch becomes a versioned hypothesis
-> deterministic replay and falsification
-> founder review
-> separately authorized rule, if warranted
```

The model output is not automatically runtime policy.

### 6.8 Hierarchical and multilevel models

Potential future use:

- partial pooling across users while preserving personal histories;
- separating task-shape effects from user-specific baselines;
- estimating uncertainty under sparse personal data;
- testing transportability from founder to cohort.

Prerequisites:

- multiple users with overlapping contexts;
- accepted plan versions;
- stable feature definitions;
- sufficient within-user and between-user support;
- exposure-aware outcome windows;
- predeclared estimand and validation.

### 6.9 Change-point and distribution-shift detection

Candidate use:

- detect when an old calibration regime no longer describes current plans;
- identify provider-coverage changes;
- identify policy or UI changes affecting observed behavior;
- reduce the weight of stale history.

A detected shift is not a claim about why the user changed.

### 6.10 Clustering

Potential use:

- discover recurring plan shapes or execution traces;
- organize hypotheses for later tests.

Prohibited use:

- turning clusters into identity archetypes without stability,
  transportability, usefulness, and harm review;
- treating cluster assignment as ground truth;
- using exposure-contaminated clusters as natural behavior.

## 7. A Candidate Honest Tree Contract

If a later plan authorizes a shadow tree, it should require:

### Inputs

- only versioned, registered deterministic features;
- only information available at the declared prediction time;
- no future leakage;
- no protected/private fields without explicit necessity and approval;
- explicit missing indicators where absence is informative;
- exposure and source features used as context, not intrinsic cost.

### Outcomes

- one registered outcome dimension per first model;
- frozen definition and window;
- censored and unknown outcomes preserved;
- raw and effective user-history outcomes analyzed separately;
- no productivity, focus, motivation, avoidance, or identity labels.

### Fitting

- shallow maximum depth;
- minimum rows, users, and active days per leaf;
- chronological training/validation split;
- no tuning on the final holdout;
- comparison against trivial, prior, and current deterministic baselines;
- bootstrap or repeated temporal stability analysis;
- split and leaf support recorded;
- no online updates in runtime.

### Output

- branch conditions;
- support and missingness;
- outcome distribution and uncertainty;
- exposure regime;
- transportability ceiling;
- abstention reason;
- method and feature versions.

### Failure

The tree remains shadow-only or is discarded when:

- branches are unstable across time;
- high-cardinality artifacts dominate;
- synthetic or retroactive paths look artificially favorable;
- performance does not beat a simple baseline;
- calibration fails;
- burden or product usefulness is not improved;
- users interpret leaves as identity judgments;
- cohort transportability fails.

## 8. Decision Support Without A Scalar Score

Candidate plan comparison can use:

### 8.0 Contract and projection discipline

There may be several decision contracts, but not several unconstrained value
systems. Each contract must:

- be the sole contract for a named decision family;
- consume a named, versioned projection from canonical `Y*`, `Q*`, and `B*`
  feature families;
- reference shared objective definitions rather than redefining them;
- consume the same active user preference snapshot as other surfaces;
- declare its action set, horizon, hard constraints, trade-offs, and fallback;
- emit a `DecisionTrace` sufficient to explain a different recommendation;
- pass cross-surface coherence fixtures.

The initial set must remain closed and small. A route, component, or experiment
does not receive a new contract merely because its implementation is separate.
A new objective or conflicting default preference requires explicit founder
review, not only a code-owner change.

A strict conflict is permitted only when the contracts declare a materially
different action set, horizon, or user preference. Otherwise Pressure Map and
recovery cannot recommend opposite scope/lateness trade-offs from equivalent
evidence.

### 8.1 Constraint filtering

Reject or mark plans that violate explicit hard constraints:

- impossible deadline;
- overlapping one-active-timer execution authority;
- missing confirmed obligation;
- unavailable time window;
- unresolved plan acceptance;
- privacy or evidence restriction.

### 8.2 Pareto dominance

A plan dominates another only when it is no worse on every registered
dimension and better on at least one, under compatible uncertainty.

### 8.3 Lexicographic preferences

Example user-owned order:

```text
1. protect tomorrow's submission deadline
2. preserve minimum scope
3. minimize displacement of already accepted work
4. prefer fewer fragmented blocks
```

This is inspectable and reversible. It should not be inferred silently.

### 8.4 Robust comparison

Prefer a plan only when its advantage survives plausible ranges, missingness,
and forecast uncertainty. Otherwise show alternatives or ask for the one
missing fact that would change the decision.

### 8.5 Resolution coverage and costly abstention

`abstain` means abstaining from an unsupported selection, not abandoning the
user. Every decision family needs a `ResolutionCoverageContract` with:

- allowed output modes and abstention reasons;
- target decision coverage by evidence-capability class;
- maximum repeated unresolved encounters;
- question, interruption, and latency budgets;
- deterministic fallback order;
- a product-readiness failure threshold.

The fallback ladder is:

```text
robust recommendation
-> bounded comparison
-> one decision-changing question
-> descriptive range or safe user choice
-> unavailable with a specific reason
```

Track at least two denominators:

```text
selection_coverage = recommend / eligible_decision_episodes
useful_coverage = (recommend + compare + ask_one + descriptive_fallback)
                  / eligible_decision_episodes
```

These are product-coverage measures, not scientific outcome measures. Reasons
must remain separated so missing evidence, preference ambiguity, privacy,
unsupported action sets, and system defects cannot collapse into one abstention
rate.

Crossing the unresolved-rate threshold disables promotion, visible automation,
or the affected surface until its evidence or design improves. It must never
force the contract to rank incomparable alternatives. This puts a real product
cost on abstention without rewarding confident fabrication.

## 9. Value-Of-Information Computations

The ontology makes a useful low-burden question possible:

> Which missing fact would most change the current plan comparison?

Candidate missing facts:

- whether an obligation is still relevant;
- minimum acceptable scope;
- a deadline's actual due time;
- whether a calendar block is movable;
- a duration range;
- whether two tasks are alternatives or both required;
- whether work already happened elsewhere.

LyraOS should prefer one decision-relevant question over a broad survey. This
can improve the deterministic free layer without adding runtime AI.

Full probabilistic value of information is not authorized by this catalogue.
It requires defended beliefs and losses. The deterministic precursor is
`ValueOfResolution` inside a registered `DecisionContract`:

```text
VoR_eligible[s](question) =
  plausible answers change at least one of:
    - the feasible action set,
    - the dominance relation,
    - the highest supported resolution mode
  AND the question fits the contract's burden budget
```

Among eligible questions, deterministic ordering prefers effects on hard
feasibility, then effects on robust dominance, then lower response burden. It
does not invent a numeric exchange rate between decision change and burden.

The canonical surface owner writes the question candidates and local decision
contract; the founder approves normative trade-offs; measurement review checks
that each answer is observable and admissible. Contracts reuse the shared
objective vocabulary and preference snapshot, so this creates several local
decision problems without creating several unrelated value systems.

There is one active contract per decision family, not one objective function
per page or endpoint. Cross-surface coherence fixtures must replay overlapping
scope/lateness cases. A contradictory strict recommendation with equivalent
evidence, horizon, action set, and preferences is a defect, not harmless local
customization.

A question is eligible only when at least one plausible answer changes the
decision and the question remains inside the surface's burden budget. If no
question qualifies, the contract follows its resolution fallback rather than
inventing information gain.

## 10. How AI May Strengthen The Deterministic Layer Later

AI is a residual semantic layer, not the plan-dynamics authority.

Potential candidate roles:

- parse ambiguous scope into editable candidate components;
- propose alternative obligation matches;
- suggest plan transformations;
- identify missing variables likely to change deterministic comparison;
- generate competing mechanism hypotheses for a deviation vector;
- translate a deterministic tree/rule path into supportive language;
- detect contradictions in an evidence packet;
- propose edge-case test fixtures;
- distill recurring AI proposals into reviewed deterministic rules.

AI may not:

- create accepted plan truth;
- decide scope or dependency truth;
- assign outcome weights;
- override clean-data admission;
- infer motivation, focus, avoidance, discipline, or identity;
- select or apply an irreversible mutation;
- create exposure truth;
- promote a claim;
- replace deterministic fallback.

## 11. Free Deterministic Layer Opportunities

Without runtime AI, the ontology can strengthen the product through:

- clearer duration and scope ranges;
- plan-shape comparison;
- source-aware calibration;
- interval-union coverage;
- explicit slack and unresolved-capacity states;
- versioned deterministic recovery transformations;
- one-question value-of-information prompts;
- shallow reviewed rule candidates distilled offline;
- scenario sensitivity and robust alternatives;
- exposure-aware calibration;
- abstention when evidence is insufficient.

The free layer should not be a deliberately weakened product. It should be the
reproducible, inspectable substrate on which optional reasoning augmentation
operates.

## 13. Evaluation Metrics For Models

Accuracy alone is insufficient.

Required model evaluation candidates include:

- denominator coverage;
- calibration by outcome range;
- uncertainty coverage;
- temporal holdout performance;
- user and context support;
- branch/rule stability;
- comparison with simple deterministic baseline;
- missingness sensitivity;
- exposure-stratified performance;
- synthetic/retroactive artifact sensitivity;
- transportability;
- abstention rate;
- resolution coverage by output mode and evidence capability;
- repeated unresolved encounters;
- cross-surface strict-conflict rate on shared fixtures;
- burden and usefulness when eventually surfaced;
- error severity by outcome dimension;
- correction and disagreement rate.

## 14. Forbidden Shortcuts

- Train on the same data used to choose thresholds and report success.
- Shuffle longitudinal rows when temporal leakage is plausible.
- Treat every task as independent.
- Learn from rows created by the prompt being evaluated without exposure
  stratification.
- Treat provider completion as execution.
- Treat missing response as rejection or failure.
- Impute unknown to zero.
- Call confidence a calibrated probability without calibration evidence.
- Call a tree leaf an archetype or user type.
- Use one scalar to hide Pareto trade-offs.
- Interpret model feature importance as causal explanation.
- let AI explanations become provenance.
