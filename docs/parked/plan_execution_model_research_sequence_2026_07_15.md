## 1. Goal

The long-term opportunity is to make LyraOS capable of learning how accepted
plan shapes relate to observable execution and work outcomes while keeping:

- canonical state deterministic;
- estimates editable;
- provenance inspectable;
- system influence exposure-complete;
- uncertainty explicit;
- user agency intact;
- claims bounded;
- models replaceable;
- runtime AI optional and subordinate.

The target sequence is:

```text
finish truth and loop reliability
-> repair label-corrupting defects
-> falsify the proposed outcome ontology on existing evidence
-> test one bounded, domain-named Sprint vertical slice
-> decide whether optional plan revision justifies durable storage
-> collect honest founder evidence
-> compute descriptive plan dynamics
-> compare simple interpretable models in shadow
-> test transportability with a cohort
-> design experiments for interventions
-> permit bounded reasoning augmentation only where residual ambiguity remains
```

## 2. What The Ontology Changes

Before this ontology, LyraOS had many useful but loosely connected
computations:

- Pressure Map workload ranges;
- duration estimates and calibration;
- task and deadline state;
- stopwatch execution;
- pause and resume prediction;
- recovery and re-entry;
- deterministic Insights;
- exposure and notification lifecycles;
- operator readiness.

They lacked one stable unit that linked:

```text
the bounded decision that became active
-> the optional accepted plan that existed then
-> the execution observed later
-> the obligations that execution advanced
-> the resulting work outcome
-> the system surfaces rendered in between
```

The ontology makes these questions well-posed:

- Did the accepted plan remain feasible?
- Which constraint changed?
- Was the deviation beneficial, harmful, or unresolved?
- Which later obligations were displaced?
- Which recovery transformation restored feasibility?
- Which plan shapes recur under similar observable conditions?
- Which missing fact would change the decision?
- Did LyraOS participate in the evidence-generating process?

It does not answer those questions by itself.

## 6. Phase 2: Analysis-Only Falsification With Current Schema

### Purpose

Attempt to disprove the proposed plan-outcome feature family before creating a
schema.

### Frozen projection

```text
latest_plan_compatibility
```

This projection must not claim historical plan quality.

### Required analyses

1. Denominator waterfall from candidate tasks to admissible episodes.
2. Distribution of plan-acceptance certainty.
3. Synthetic immediate-start versus ordinary-plan comparison.
4. Retroactive `planned = executed` versus ordinary-plan comparison.
5. Exact-adherence/poor-outcome case search.
6. Large-deviation/good-outcome case search.
7. Raw versus corrected versus clean projection sensitivity.
8. Summed task occupancy versus interval-unioned session time.
9. Exposure-known, known-unexposed, and unknown strata.
10. Estimate-source and provider-source strata.
11. Scalar-weight rank-reversal simulation.
12. Missingness and censoring inventory.

### Expected result

The likely outcome is that some descriptive dimensions survive while a
universal scalar cost fails.

### Gate

Proceed to a schema decision only when:

- the surviving dimensions describe real product questions;
- plan-version loss is a demonstrated limitation rather than architectural
  preference;
- current product use produces enough accepted planning events;
- the additional capture burden is acceptable;
- the result cannot be achieved safely from existing append-only evidence.

### Stop condition

If current evidence shows little accepted planning, low outcome completeness,
or no product use for the vector, stop. Improve product usefulness and capture
rather than creating schema.

## 7. Phase 3: Bounded Sprint And Optional Plan-Revision Decision

### Purpose

Decide whether one bounded Sprint projection and optional durable plan-
revision evidence are warranted. Do not persist a generic `DecisionEpisode` or
build a generalized episode platform.

### Required decision packet

- demonstrated user decision blocked by mutable Task rows;
- episode opening event, bounded obligation set, decision-time snapshot,
  closure policy, and allowed relations;
- proof that opening, membership, and closure are reconstructible without
  hindsight;
- why the same value cannot be obtained by composing current append-only
  evidence;
- proposed authority owner;
- acceptance semantics;
- draft versus accepted versus superseded semantics;
- versioning and idempotency;
- task/deadline/calendar linkage;
- plan-bundle boundary, if any;
- optional outcome-criterion and contribution-attribution burden;
- raw/effective/clean projections;
- export and deletion coverage;
- retention and privacy;
- backfill policy;
- unknown and legacy-row policy;
- rollback and migration reversal;
- fresh-database migration proof;
- browser paths that create or supersede a version.

### Required founder choice

Possible choices:

- no schema; keep latest-plan compatibility only;
- one narrow founder-only Sprint projection assembled from current evidence;
- append-only task plan-revision events;
- accepted-plan snapshot per task;
- accepted plan bundle with member references;
- another minimal design demonstrated by evidence.

No option is preferred by this parked document.

### Candidate pre-cohort vertical slice

If later approved, the smallest useful slice is:

```text
Pressure Map
-> user selects obligations
-> Start Sprint
-> existing stopwatch execution
-> pause, switch, and recovery evidence
-> lightweight outcome reconciliation
```

The bounded Sprint projection would snapshot selected obligations, accepted effort
range, protected deadline, minimum and optional scope, active preference,
activation, proposal provenance, and projection/surface versions. Closure
would ask which obligations moved: `complete`, `partial`, `untouched`,
`replaced`, `irrelevant`, or `unknown`. Shared or approximate time attribution
would remain explicit.

Only one projection (`sprint_execution.v1`), one Pressure Map/Sprint decision
contract, one computed debug trace, and one deterministic
`ValueOfResolution` question family may be considered. A universal cost
function, persisted `DecisionEpisode`, generalized episode platform, task graph, multiple decision
contracts, runtime learned ranking, automatic mutation, or probabilistic VoI
remain parked.

### Fragment discipline

Do not create a universal semantic-field registry, fragment registry, or
authority-promotion ladder before this slice. The approved implementation plan
must instead contain one local Data Delta classifying each proposed item as
`CREATE`, `REFERENCE`, `SNAPSHOT`, `DERIVE`, or `FORBID_COPY`, with its existing
owner and immediate consumer.

The first Sprint candidate must also resolve these questions before migration:

- one active execution boundary must be separate from pending reconciliation;
- supersession is bounded domain lineage and cannot fork in founder v1;
- no automatic expiry exists without an executable expiry policy;
- clarification is not forced into execution-progress closure semantics;
- activation is one idempotent product command over canonical database truth
  with convergent Redis recovery;
- closure evidence cannot become a second Task, Deadline, or estimate
  authority;
- no hidden Sprint Task is created.

See
`docs/audits/founder_core_loop_episode_fragment_stress_test_2026_07_16.md`
and
`docs/parked/founder_core_loop_work_sprint_candidate_plan_2026_07_16.md`.

### Stop condition

If two plausible authority owners or units of analysis remain, stop for founder
decision. Do not hide the ambiguity inside a generic event table.

- execution coverage;
- outcome completeness;
- plan-version coverage;
- exposure completeness;
- correction and recovery rates;
- missingness;
- active-day support;
- burden and annoyance feedback.

### Stop condition

If instrumentation changes behavior enough that the product becomes annoying
or QA-dependent, reduce burden. Do not continue solely to collect data.

### Candidate transformations

## 11. Phase 7: Interpretable Shadow Models

### Purpose

Test whether data-driven conditional partitions add value beyond rules and
descriptive distributions.

### First model constraints

- one registered outcome dimension;
- shallow depth;
- versioned deterministic features;
- chronological train/calibration/holdout split;
- minimum rows, active days, and users per leaf;
- exposure-aware analysis;
- no online updates;
- no user-visible output;
- no runtime mutation;
- complete model card and deletion path.

### Candidate baselines

- global median or rate;
- category/time deterministic cell;
- current personal calibration heuristic;
- conservative research prior;
- no-model abstention.

Before any decision-support output becomes visible, freeze a small closed set
of decision families. Each family must have:

- one named consumer projection over canonical feature families;
- shared objective definitions and an explicit preference snapshot;
- one versioned decision contract;
- a resolution-coverage contract with useful non-ranking fallbacks;
- a decision trace;
- cross-surface fixtures for overlapping scope, lateness, burden, and recovery
  cases.

If equivalent contracts produce opposing strict recommendations without a
declared difference in horizon, action set, or user preference, decision
support remains parked. If abstention exceeds the declared usefulness boundary,
the surface returns to comparison, one-question, or descriptive mode; the
coverage target does not compel a ranking.

### Candidate models

- shallow regression tree;
- conditional inference tree;
- survival tree/model for censored time;
- reviewed rule list;
- simple regularized regression where more stable than a tree.

### ID3 decision

Classic ID3 may be used as an educational or exploratory baseline for a truly
categorical outcome. It should not be the production default because plan
dynamics include continuous, censored, missing, and multi-objective outcomes.

### Gate

The model must:

- beat a simple baseline on untouched temporal data;
- remain calibrated enough for its intended wording;
- preserve support and uncertainty;
- remain stable across reasonable analysis choices;
- abstain under weak support;
- avoid synthetic and exposure artifacts;
- add a named product or research benefit.

### Stop condition

If the model only creates more complicated thresholds without reducing error,
uncertainty, burden, or decision regret, keep the deterministic baseline.

## 13. Phase 9: Intervention Experiment Design

### Purpose

Identify whether a plan or recovery intervention changes a proximal outcome.

### Prerequisites

- stable decision point;
- availability and eligibility definition;
- complete option set including no treatment;
- assignment probabilities;
- burden and quiet-hour limits;
- proximal outcome and window;
- carryover treatment;
- exposure and lifecycle truth;
- missingness policy;
- pre-registered analysis;
- pause/kill authority;
- founder and ethical approval as applicable.

### Candidate designs

- founder N-of-1 with temporal randomization;
- feasibility study;
- micro-randomized trial;
- later optimization design only after component evidence exists.

Ordinary dogfood, deterministic 1-in-7 labels, and user-id parity are not
experiments.

### Stop condition

Without logged assignment probability and complete exposure/outcome windows,
no causal intervention claim is allowed.

## 14. Phase 10: Bounded Reasoning-Runtime Augmentation

### Purpose

Use a reasoning runtime only for residual semantic ambiguity after deterministic
and statistical methods have first refusal.

### Candidate roles

- parse ambiguous scope into editable candidates;
- suggest competing obligation matches;
- propose missing variables;
- propose plan transformations;
- generate competing explanations for an observed deviation vector;
- produce supportive wording grounded in registered literature and evidence;
- summarize an evidence packet;
- propose deterministic edge-case fixtures.

### Required path

```text
admitted and minimized EvidencePacket
-> ReasoningRuntimeContract
-> replaceable adapter
-> structured candidate
-> deterministic validation
-> ClaimCompiler and surface policy
-> authenticated browser render
-> explicit user action through canonical authority
```

### Forbidden ownership

The reasoning layer does not own:

- accepted plan truth;
- task, deadline, provider, or execution state;
- duration weights or outcome utility;
- clean-data admission;
- experiment randomization;
- exposure truth;
- irreversible mutation;
- claim promotion;
- psychological or causal attribution.

### Gate

Reasoning augmentation must beat the complete deterministic baseline on a
predefined residual ambiguity set without reducing trust, privacy, latency,
reproducibility, or fallback quality.

## 15. What Can Be Learned At Each Stage

| Stage | Defensible learning | Not defensible |
| --- | --- | --- |
| Current schema | Latest-plan descriptive mismatch | Historical plan quality |
| Accepted plan versions | Revision-aware plan dynamics | Causal intervention effect |
| Founder repeated use | Founder-specific conditional patterns | Student/cohort default |
| Cohort repeated use | Transportability and personal variation | Cause of behavior |
| Randomized decision points | Proximal treatment effect under protocol | Universal long-term benefit |
| Reasoning augmentation | Semantic candidate quality | Model explanation as truth |

## 16. Anti-Scope-Inflation Rules

- The ontology is not a request to implement every possible metric.
- The computation catalogue is not a backlog.
- A model is not justified because a feature exists.
- A schema is not justified because a concept is elegant.
- AI is not justified because an edge case is hard.
- Governance is downstream of product usefulness and truthful exposure; it is
  not the product.
- Every phase must solve a demonstrated current problem.
- Adjacent discoveries become issues, not automatic work.
- Three consecutive additions that improve neatness but not usefulness,
  validity, or reversibility stop the sequence.
- If a simple deterministic method performs as well, keep it.

## 17. Decision States

Each promoted phase ends in one of:

- `promising`: useful enough to justify the next bounded question;
- `weak`: mechanism works but product/research value is insufficient;
- `blocked`: evidence or authority prerequisite is missing;
- `invalidated`: falsification defeats the proposed construct;
- `not_tested`: insufficient real use;
- `retired`: no longer worth maintaining.

`inconclusive` is evidence, not failure.

## 18. Final Principle

```text
The ontology should make LyraOS more useful because it helps the system compare,
repair, and learn from plans.

It should not make LyraOS more authoritative merely because it can calculate
more things.
```
